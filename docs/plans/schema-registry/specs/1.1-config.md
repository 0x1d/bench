# Task 1.1: Config

## Context

Add schema registry config support. Schemas are defined in `resources.schemas[]` with `id`, `label`, `type`, and `source.path`. Valid types: `openapi`, `asyncapi`, `json-schema`. Path is relative to config directory (same as `openapiSpec` for REST).

Reference: [plan.md](../plan.md) § 4.2, § 6.1.

## Steps

1. **Define SchemaEntry struct** in `api/internal/config/config.go`:
   - `ID string` (yaml: `id`)
   - `Label string` (yaml: `label`)
   - `Type string` (yaml: `type`) — one of `openapi`, `asyncapi`, `json-schema`
   - `Source struct { Path string }` (yaml: `source`) — `Path` for `path` key

2. **Add Schemas to ResourcesConfig**:
   - Add `Schemas []SchemaEntry` (yaml: `schemas`) to `ResourcesConfig` struct

3. **Add SchemaEntries()**:
   - Returns `[]SchemaEntry` from `cfg.Resources.Schemas`
   - Skip entries with empty `ID` or `Type`
   - Default `Label` to `ID` when empty
   - Use `ReadConfig()`; return empty slice on error

4. **Add SchemaByID(id string)**:
   - Returns `*SchemaEntry` or nil
   - Iterate `SchemaEntries()` and match by `ID`

5. **Add validation in validateConfig()**:
   - Unique `id` per schema (no duplicates)
   - `id` required
   - `type` required and must be one of: `openapi`, `asyncapi`, `json-schema`
   - `source.path` required when `type` is set
   - Return `fmt.Errorf` with clear message on failure

6. **Update config.example.yaml**:
   - Add `schemas` section under `resources` with example:
     ```yaml
     schemas:
       - id: petstore-api
         label: Petstore API
         type: openapi
         source:
           path: ./workspace/rest/petstore.json
     ```

## Acceptance Criteria

- [ ] `SchemaEntries()` returns configured schemas from valid config
- [ ] `SchemaByID("petstore-api")` returns the entry when present

- [ ] Config with invalid type `foo` fails validation
- [ ] Config with empty `source.path` fails validation
