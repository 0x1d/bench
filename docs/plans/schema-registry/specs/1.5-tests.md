# Task 1.5: Tests

## Context

Add unit tests for the schema service and integration tests for the schema API handlers. Use table-driven tests for Go. Use existing test patterns from `api/internal/service/rest` and `api/internal/handler` if present.

## Steps

1. **Create test config fixture**:
   - `api/internal/config/testdata/schema-registry-config.yaml` — minimal config with `resources.schemas` and a schema file path
   - Include a minimal valid OpenAPI spec in testdata (e.g. `testdata/petstore-minimal.json`) so `Content()` can read it
   - For path traversal test: use a path like `../` or `../../etc/passwd` in a config fixture; assert `Content()` returns error

2. **Unit tests for schema service**:

   Create `api/internal/service/schema/service_test.go`:
   - Set `BENCH_CONFIG` to testdata config path (or use helper to load test config)
   - Test `List()` with config containing schemas — returns correct count and IDs
   - Test `List()` with empty config — returns empty slice
   - Test `Get(id)` with valid id — returns resource
   - Test `Get(id)` with invalid id — returns ErrSchemaNotFound
   - Test `Content(id)` with valid id and existing file — returns bytes
   - Test `Content(id)` with invalid id — returns ErrSchemaNotFound
   - Test `Content(id)` with path traversal in path — returns error

3. **Integration tests for API**:

   Create or extend `api/internal/handler/schema_test.go`:
   - Test `GET /api/schemas` — status 200, JSON body `{ "schemas": [...] }`
   - Test `GET /api/schemas/{id}` — 200 when exists, 404 when not
   - Test `GET /api/schemas/{id}/content` — 200 with body when exists
   - Test `POST /api/schemas` — 405 if route exists, or ensure no route

   Use `httptest.ResponseRecorder` and `http.NewRequest`; inject or mock schema service if needed.

4. **Ensure tests pass** with existing config structure. If schema config is empty in tests, `List()` should return empty slice; handlers should still return 200 with empty array.
