# Task 1.3: Trigger Validation

## Context

Add validation logic for trigger entries in config.yaml. Reuse patterns from `validateConfig()` for uniqueness and required field checks.

Reference: [plan.md](../plan.md) § 7.

## Steps

1. **Add validation for `TriggerEntry`** in `validateConfig()`:
   - `id` required (check in loop)
   - `flow` required (check in loop)
   - `type` required and must be one of: `webhook`, `schedule`, `alert`, `http`, `notification`
   - Unique `id` per entry (no duplicates)
   - `config` required with at least `pipeline` field

2. **Add trigger-specific validation**:
   - For `webhook`: check `config.pipeline` exists
   - For `schedule`: check `config.schedule.cron` is valid
   - For `alert`: check `config.alert.source` and `config.alert.condition` exist
   - For `http`: check `config.http.url` exists

3. **Add helper validation functions**:
   - `isValidTriggerType(t TriggerType) bool`
   - `validateScheduleCron(cron string) error`
   - `validatePipelineRef(ref string) error`

4. **Return clear error messages**:
   - Include array index in error: `flowpipe_triggers[0]: type is required`
   - Include field name in error: `flowpipe_triggers[1].config.schedule.cron is invalid`

## Acceptance Criteria

- [ ] Config with duplicate trigger id fails validation
- [ ] Config with missing `type` fails validation
- [ ] Config with invalid type `foo` fails validation
- [ ] Config with missing `flow` fails validation
- [ ] Config with invalid cron syntax fails validation
