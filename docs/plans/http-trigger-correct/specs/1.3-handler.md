# Task 1.3: Update API handlers and webhook URL endpoint

## Context

The API handlers in `flow.go` have `normalizeTriggerConfig` which maps flat UI config keys into nested Go structs. It currently handles both `webhook` and `http` flat keys. The webhook URL endpoint (`HandleTriggerWebhookURL`) works for both types but conceptually should only apply to `http` triggers now. The `TestTrigger` function also has special handling for webhook triggers that needs updating.

## Steps

1. **Update `normalizeTriggerConfig` in `api/internal/handler/flow.go`**:
   - Remove `webhook` nested object handling
   - Update `http` handling to map `args` and `executionMode` flat keys
   - Remove any references to `url`, `method`, `body` flat keys for HTTP type

2. **Update `HandleTriggerWebhookURL` and `HandleRootTriggerWebhookURL`**:
   - These endpoints should work for `http` type triggers (they already generate the correct URL)
   - No code changes needed — they work by trigger ID, not type
   - Update any error messages or comments referencing "webhook trigger"

3. **Update `TestTrigger` in `api/internal/service/flow/trigger.go`**:
   - Change `triggerType == model.TriggerTypeWebhook` to `triggerType == model.TriggerTypeHTTP`
   - The test payload generation remains the same (simulating an inbound webhook POST)

## Files

| Path | Action |
|------|--------|
| `api/internal/handler/flow.go` | Modify — normalizeTriggerConfig |
| `api/internal/service/flow/trigger.go` | Modify — TestTrigger webhook→http |

## Acceptance Criteria

- [ ] `normalizeTriggerConfig` has no webhook handling
- [ ] `normalizeTriggerConfig` maps `args` and `executionMode` for HTTP type
- [ ] `TestTrigger` checks for `TriggerTypeHTTP` not `TriggerTypeWebhook`
- [ ] Webhook URL endpoint still works for HTTP triggers
