# Task 2.2: Phase 2 — REST Service

**Task ID**: 2.2  
**Phase**: 2 — REST Migration to Schema Registry  
**Prerequisites**: 2.1 (Config)

## Context

Update REST service `Spec(id)` to resolve spec via schema registry when `schemaId` is set. Fall back to file read from `openapiSpec` when `schemaId` is empty. Add `SchemaID` to `RestResource` API response for UI.

Reference: [plan.md](../plan.md) § 5 Phase 2.

## Steps

1. **Add SchemaID to RestResource** in `api/internal/model/rest.go`:
   - `SchemaID string` (json: `schemaId,omitempty`)

2. **Update REST service List()** in `api/internal/service/rest/service.go`:
   - When building `model.RestResource`, set `SchemaID: e.SchemaID`

3. **Update REST service Spec(id)**:
   - Get entry via `GetEntry(id)`
   - If `entry.SchemaID != ""`:
     - Create schema service (or inject): `schema.NewService()`
     - Call `schemaSvc.Content(entry.SchemaID)`
     - Return result (or error)
   - Else if `entry.OpenAPISpec != ""`:
     - Existing logic: resolve path, read file
   - Else:
     - Return `ErrSpecNotFound`

4. **Avoid circular imports**: Schema service must not import REST service. REST service imports schema service. Consider passing schema service as dependency or using a simple function to get content by schema id.

## Acceptance Criteria

- [ ] REST resource with schemaId returns spec from schema registry
- [ ] REST resource with only openapiSpec returns spec from file (unchanged behavior)
- [ ] REST resource with both: schemaId wins
- [ ] REST resource with neither: ErrSpecNotFound
- [ ] List() includes schemaId in response when set
