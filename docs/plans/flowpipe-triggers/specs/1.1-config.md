# Task 1.1: Trigger Config Model

## Context

Define the data structures for Flowpipe triggers configuration. This includes trigger type constants, TriggerEntry struct (for config.yaml), TriggerConfig (type-specific config), and TriggerState (for API responses).

Reference: [plan.md](../plan.md) § 3.1, § 5.1.

## Steps

1. **Create `api/internal/model/trigger.go`**:
   - Define `TriggerType` type and constants:
     ```go
     type TriggerType string
     const (
         TriggerTypeWebhook TriggerType = "webhook"
         TriggerTypeSchedule TriggerType = "schedule"
         TriggerTypeAlert TriggerType = "alert"
         TriggerTypeHTTP TriggerType = "http"
         TriggerTypeNotification TriggerType = "notification"
     )
     ```
   - Define `TriggerConfig` struct with type-specific fields:
     - `Description string`
     - `Pipeline string` (pipeline reference)
     - `Webhook *WebhookConfig`
     - `Schedule *ScheduleConfig`
     - `Alert *AlertConfig`
     - `HTTP *HTTPConfig`
     - `Notification *NotificationConfig`
   - Define `TriggerEntry` struct (for config.yaml):
     - `ID, Label, Workspace, Flow, Type, Config`
   - Define `TriggerState` struct (for API responses):
     - `ID, Label, Flow, Type, Workspace, Enabled`
     - `Config, LastRun, NextRun, Status`

2. **Update `api/internal/config/config.go`**:
   - Add `TriggerEntry` to config (same pattern as `SchemaEntry`)
   - Add `Triggers []TriggerEntry` to a new `FlowpipeConfig` struct or add to `FlowsConfig`
   - Keep `FlowsConfig` unchanged and add `FlowpipeTriggers` as a sibling to `Flows`, `Agent`, etc.

3. **Add `TriggerEntries()` function**:
   - Returns `[]TriggerEntry` from config
   - Skip entries with empty `ID` or `Flow`
   - Default `Label` to `ID` when empty

4. **Add `TriggerByID(id string)` function**:
   - Returns `*TriggerEntry` or nil
   - Iterate `TriggerEntries()` and match by `ID`
