# Task 2.1: Config (REST schemaId)

## Context

Add `schemaId` to REST resources so they can reference a schema from the registry instead of (or in addition to) `openapiSpec`. Precedence: `schemaId` wins over `openapiSpec`.

Reference: [plan.md](../plan.md) § 4.2, § 6.3.

## Steps

1. **Add SchemaID to RestEntry** in `api/internal/config/config.go`:
   - `SchemaID string` (yaml: `schemaId,omitempty`)

2. **Add validation in validateConfig()** for REST entries:
   - If `rest.SchemaID` is non-empty:
     - Call `SchemaByID(rest.SchemaID)` — must return non-nil
     - The schema's `Type` must be `openapi`
   - Error message: e.g. `resources.rest[%d].schemaId %q references non-existent or non-openapi schema`

3. **Document precedence** in code comment: when both schemaId and openapiSpec are set, schemaId is used for spec resolution.

## Files

| Path | Action |
|------|--------|
| `api/internal/config/config.go` | Modify |

## Acceptance Criteria

- [ ] RestEntry has `schemaId` field; config parses it
- [ ] Config with valid schemaId (openapi type) passes validation
- [ ] Config with schemaId pointing to non-existent schema fails validation
- [ ] Config with schemaId pointing to asyncapi schema fails validation (REST requires openapi)
